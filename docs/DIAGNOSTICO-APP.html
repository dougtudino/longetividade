2Login Admin (painel de gestão)

Quem entra no painel /admin/* são os operadores do negócio (você + Bárbara). Não tem cadastro aberto — é por seed/convite.

Fluxo de autenticação

1
POST /api/admin/auth/login
Recebe {email, password}
2
Busca AdminUser
Por email (lowercased)
3
bcrypt.compare
Confronta com passwordHash salvo
4
signAdminToken (HMAC)
Body base64url + HMAC-SHA256 (TTL 7 dias)
5
Cookie httpOnly
admin-token, sameSite=lax, secure em prod
6
middleware.ts
Cada request /admin/* e /api/admin/* revalida o token

Como criar um Admin novo

  • Via UI: /admin/admins (página existente) tem CRUD com hashing bcrypt.
  • Via seed inicial: POST /api/admin/auth/seed com header seedKey=ADMIN_SEED_KEY e payload {users:[{email,name,password,role}]}. Roda prisma db push antes de inserir — usado no bootstrap do banco.
  • Convite por token: o schema tem AdminUser.inviteToken + inviteUsed, mas o fluxo de convite ainda é manual (não há rota dedicada de "aceitar convite").
  • Google OAuth: o admin tem campo googleId e /api/auth/google/start?context=admin existe — vincula um Google account a um AdminUser já cadastrado.
Token JWT caseiro (lib/admin-token.ts) 
// HMAC-SHA256 com secret = JWT_SECRET || NEXTAUTH_SECRET || ADMIN_PASSWORD || fallback
// Payload: { adminId, email, name, role, exp } base64url + assinatura
// TTL: 7 dias. Cookie httpOnly, sameSite=lax.
// Verificação faz comparação constant-time do HMAC.
Risco moderado: o secret usa fallback hardcoded ("longetividade-admin-fallback-secret") se nenhuma env var estiver setada. Verificar em produção que JWT_SECRET ou NEXTAUTH_SECRET existe no Railway.

3Login do usuário (App VIP)

É o login da cliente que comprou o plano VIP. UI mobile-first em /app/login e /app/cadastro.

Três cenários no POST /api/app/auth

CenárioComportamento
A. AppUser existe e tem passwordHashExige senha. Se ausente, devolve requiresPassword:true e a UI revela o campo de senha.
B. AppUser existe sem senha (legado)Aceita email-only — basta digitar o e-mail que entrou na compra.
C. AppUser não existeBusca Order aprovada com esse e-mail. Se achou, cria o AppUser na hora com plan herdado e libera acesso (fallback do webhook).

Cadastro de senha — POST /api/app/register

  • Exige Order VIP aprovada com o mesmo e-mail (status=approved).
  • Faz upsert em AppUser e gera passwordHash bcrypt (cost 10).
  • Seta cookies app_token (TTL 1 ano!) + app_email (legacy).

Google OAuth

Botão "Continuar com Google" em /app/login aponta para /api/auth/google/start?context=app. O callback grava googleId no AppUser e seta o cookie.

Demo login (admin testando o app)

Botão "Entrar como Admin" em /app/login chama /api/app/demo-login. Útil pra Bárbara experimentar o app sem comprar.

Atenção: o cookie app_token tem TTL de 1 ano. Se o usuário compartilhar o dispositivo, a sessão dura muito. Ok pra app pessoal, mas vale revisar.

Onboarding (primeira entrada)

/app/onboarding tem 5 passos: nome+objetivo → dados corporais → meta de peso → desafios → resumo. Salva em AppProfile com onboardingDone=true. Roteamento em /app/page.tsx redireciona automaticamente:

profile?.onboardingDone === true   →  /app/home
profile?.onboardingDone === false  →  /app/onboarding
sem sessão                          →  /app/login

4Compra Hotmart → liberação de acesso

Esse é o caminho crítico. Tem 3 vias de redundância pra garantir que o cliente nunca fique sem acesso.

Via 1 — Webhook em tempo real

1
Hotmart envia POST
/api/webhooks/hotmart com header x-hotmart-hottok
2
Valida secret
Header == HOTMART_WEBHOOK_SECRET (AppSetting ou env)
3
Determina plano
offer_code → basico / completo / vip (fallback por valor)
4
Cria Order
status=approved, gera downloadToken + tokenExpiresAt(72h)
5
Se VIP, claim slot
claimVipSlot() incrementa AppVipSlot.usedSlots + cria AppUser
6
Email de entrega
Brevo · "Seu ebook chegou!" com link /api/download?token=...
7
Meta CAPI
Envia Purchase server-side, event_id = purchase_HP{txn} pra dedup com Pixel

Via 2 — Sincronização via Sales API (sync diário/manual)

Se o webhook falhar (caiu, hottok mudou, etc.), o dashboard tem botão "Sincronizar Hotmart" que chama POST /api/admin/sync-hotmart. Há também cron diário /api/cron/sync-hotmart:

  • OAuth client_credentials → token cached 1h.
  • GET /payments/api/v1/sales/history paginado (até 30 dias).
  • Upsert por hotmartTransactionId (chave única) — atualiza status (refund/chargeback/canceled).

Via 3 — Fallback no login

Se nenhuma das vias acima rodou, o próprio /api/app/auth cria o AppUser na hora se achar uma Order aprovada com aquele e-mail. defesa em profundidade

Identificação por offer code

Defaults em AppSetting (editável em /admin/configuracoes):

  • HOTMART_OFFER_BASICO → zxq5tgew
  • HOTMART_OFFER_COMPLETO → uzvdkzkf
  • HOTMART_OFFER_VIP → h84hak4e

Se o offer_code não bater (campanha nova sem cadastro), cai pro fallback por valor: ≥ R$ 97 = VIP, ≥ R$ 67 = completo, senão básico.

Como a cliente recebe o acesso:
  1. Compra na Hotmart com o e-mail X.
  2. Recebe e-mail "Seu ebook chegou!" com botão BAIXAR MEU EBOOK AGORA (link com token).
  3. Se for VIP, recebe também o e-mail de convite ao App: "Seu acesso VIP está liberado!" com botão Acessar meu App VIP → /app/login.
  4. Ela entra com o e-mail da compra (email-only no primeiro acesso) ou cria senha em /app/cadastro.
  5. Faz o onboarding de 5 passos e cai no /app/home.
Bug histórico/risco: o webhook hoje cria a Order sem checar duplicata (não usa hotmartTransactionId no momento do INSERT). Em caso de retry da Hotmart pode duplicar — a sync diária reconcilia, mas vale ajustar pra upsert também no webhook.

Como a venda é monitorada — Dashboard (/admin/dashboard)

  • KPIs em tempo real: Receita hoje / mês, vendas hoje, taxa de conversão (orders / (orders + abandoned)), abandonos do dia.
  • Receita 30d em gráfico SVG nativo (sem libs).
  • Distribuição por plano com barras proporcionais.
  • Funil: Visitantes → Iniciaram checkout → Compraram.
  • Últimas vendas (tabela com status colorido).
  • Botão "Sincronizar Hotmart" que reconcilia via Sales API.
  • App VIP block: total de usuárias, ativas hoje, total checkins, vagas restantes.
  • Maya Chat: assistente conversacional com contexto carregado (precisa ANTHROPIC_API_KEY).

Como o cliente é monitorado — Clientes App (/admin/app-users)

Lista todos AppUsers VIP com métricas reais:

  • Filtros: Todos · Ativos (checkin nos últimos 7d) · Inativos · Sem onboarding.
  • Busca: por e-mail ou nome do perfil.
  • Em cada linha: avatar com iniciais, e-mail, 🔥 streak em dias consecutivos, Nv + XP, kg perdidos vs. total de pesagens, "último checkin" relativo (hoje/ontem/Xd atrás).
  • Estatísticas no topo: Total VIP · Ativas (7d) · Onboarding feito · Streak médio · Peso perdido total.
  • Clique no card → /admin/app-users/[id] com detalhe completo (pesos, água, humor, medidas, desafio, conquistas, últimos checkins, dados do pedido).

Liberar acesso manualmente (caso o webhook falhe)

  • POST /api/admin/grant-vip body {email, name?, sendInvite?:true}:
    1. Se já tem AppUser → reenvia convite por e-mail.
    2. Senão cria Order VIP "manual" (R$ 97 fictícios) + AppUser + envia convite.
  • Há UI ligada a essa rota em /admin/app-users (botão "Conceder VIP").

8Manual de uso (resumo)

Para a cliente final (VIP)

  1. Receba o e-mail "Seu acesso VIP está liberado!" e clique em Acessar meu App VIP.
  2. Em /app/login, digite o e-mail da compra. (Se a conta já tem senha, ela aparece.)
  3. Opcional: clique em "Continuar com Google" pra vincular.
  4. Preencha o onboarding (5 passos · 3 min): nome, objetivo, peso/altura/idade, meta, desafios.
  5. Use a Home pra check-in diário: marque hábitos, registre água/peso/humor.
  6. Faça o Desafio 21 Dias pra criar tração rápida.
  7. Veja Progresso pra acompanhar peso/medidas no tempo.

Para o operador (Doug/Bárbara)

  1. Login admin: /admin/login com e-mail + senha (bcrypt).
  2. Rotina diária: abra /admin/dashboard e clique Sincronizar Hotmart (puxa vendas reais).
  3. Monitorar clientes VIP: /admin/app-users → ver quem está ativo, quem não fez onboarding.
  4. Cliente reclamou que não recebeu acesso? Use /admin/app-users → "Conceder VIP" (POST /api/admin/grant-vip). Envia convite por e-mail.
  5. Editar imagens da LP: /admin/lp-assets (sobe pra R2 + invalida cache).
  6. Trocar segredos de integração: /admin/configuracoes (AppSetting). Não mexe em env Railway.
  7. Ver IA/automação: /admin/social-media (Luna), /admin/video-intelligence, /admin/agents/gaia.
  8. Pendências do sistema: /admin/setup mostra checklist auto-detectado.

10Pontos de risco & oportunidades de melhoria

🔴 Críticos

ItemRiscoSugestão
JWT com secret fallback hardcoded Se nenhuma env (JWT_SECRET, NEXTAUTH_SECRET, ADMIN_PASSWORD) estiver setada no Railway, todos os tokens podem ser forjados. Falhar explicitamente quando não houver secret. Auditar Railway agora.
Webhook Hotmart cria Order sem upsert Retry da Hotmart duplica Order. hotmartTransactionId é único, mas o webhook hoje nem grava esse campo no INSERT. Trocar create por upsert({where:{hotmartTransactionId}}) e gravar o txn id.
Cookie legacy app_email não-assinado Middleware aceita app_email como fallback — qualquer cookie com formato "email" valida. Definir cutoff e remover. Já existe app_token assinado pronto.
Crons de e-mail não agendados em prod Nurture pós-compra D+1/D+7/D+21 não dispara — perda de engajamento e refund preventivo. Configurar cron-job.org com ?secret=$CRON_SECRET em todas as rotas de e-mail.

🟢 Melhorias de experiência

  • PWA real: adicionar manifest + service worker — a UI já é mobile-first. App "instalável" no celular reduz fricção (Bárbara pede isso há tempos).
  • Push notification (web push) pra streak — empurra check-in no celular às 21h se o usuário não fez ainda hoje.
  • Magic link no login em vez de email-only direto — mais seguro e UX igual.
  • Logs estruturados (Pino/Winston) saindo pra um destino (Logtail/Axiom/Sentry) — hoje só console.error.
  • Health check de integrações na /admin/setup: já existe parcial; bom expandir pra Hotmart/Brevo/R2/Meta com latência e último sucesso.
  • Auditoria de ações admin: hoje não há tabela de audit log. Quem deu VIP manual? Quando trocou setting? Vale criar AdminAuditLog.
  • 2FA em /admin/login (TOTP) — é painel financeiro, vale o esforço.
  • Versionar receitas/desafios: hoje vivem no banco mas sem CMS/versão. Bom planejar como atualizar conteúdo sem perder favoritos.
  • Export CSV em /admin/app-users — útil pra Bárbara mandar mensagem WhatsApp em lote.
  • UI de "concedido manualmente": hoje a Order criada por grant-vip tem amount=R$97 fictício. Marcar como source:"manual-grant" pra não inflar receita real.
